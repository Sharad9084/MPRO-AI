TAG-mPRO Project Explanation
============================

Project name:
TAG-mPRO - Marketing Performance Reconciliation and Optimisation

Purpose:
This application helps an auditor validate marketing campaign spending by uploading multiple business files, converting them into structured tables, and reconciling values across Program, PR, PO, schedule, invoice and monitoring data.

Current application flow:
1. Sign in screen
2. Role selection screen
3. Auditor sign in
4. Accounting workspace
5. Upload files
6. Review source tables
7. Use filters and configurable columns
8. Review reconciliation issues
9. Save the reconciliation case

Login:
Username: auditor@mpro.com
Password: Auditor@2026

Security:
Accounts are now created through the backend API.
Passwords are not stored as plain text. The backend stores a password salt and PBKDF2-SHA256 password hash.
Password rules require at least 8 characters, uppercase, lowercase, number and special character.
Repeated failed sign-in attempts lock the account temporarily.
Sign in creates a server-side session token.
Sign off revokes the active session token.
Reconciliation cases require a valid session token to load or save through the API.

Roles:
1. Advertiser Login - planned for next phase
2. Agency Login - planned for next phase
3. Auditor Login - active now

Main upload inputs:
1. Media Schedule
2. Agency Invoice
3. 3rd Party Invoice
4. 3rd Party Monitoring Report
5. PO

Logical tables in the app:
1. Program / Campaign
   Top-level campaign data such as budget, campaign ID, campaign name and manager.

2. PR
   Purchase requisition level data. One campaign/program can have multiple PR numbers.

3. PO
   Purchase order data. One PR can have multiple PO numbers. Total PO amount should not exceed PR amount.

4. Media Schedule
   Planned campaign execution data such as channel, program, date, air time, spots and planned amount.

5. Agency Invoice
   Agency invoice data with invoice number, invoice date, PO/PR references, campaign and cost values.

6. 3rd Party Invoice
   External vendor invoice data. For TV this means broadcaster invoice, for print publisher invoice, and for other media a 3rd party vendor invoice.

7. 3rd Party Monitoring Report
   Proof/monitoring data. For TV this can be AdEx or BARC, for print tear sheet copies, and for digital platform/3rd party monitoring/DCM reports.

8. Reconciliation
   Calculated table that compares the above tables and highlights missing data, amount mismatches and budget hierarchy issues.

Important business hierarchy:
Program/Campaign is the top level.
One Program/Campaign can have many PR numbers.
One PR number can have many PO numbers.
All PR amounts together should stay within the campaign/program budget.
All PO amounts under a PR should stay within that PR amount.

Current reconciliation checks:
- PO missing
- Media schedule missing
- Agency invoice missing
- 3rd party invoice missing
- Monitoring missing
- PR amount exceeds program budget
- PO amount exceeds PR amount
- PO vs agency amount mismatch
- Schedule vs 3rd party invoice amount review
- Agency vs 3rd party invoice amount review

Column customization:
The Columns button lets the user show or hide columns for the active table. The grid also supports sorting, resizing and reordering columns.

Storage behavior:
The frontend uses API_BASE = http://127.0.0.1:8787.
If the backend API is running and the user is signed in, Save sends data to the local backend server.
The local SQLite backend stores data in backend/mpro_reconciliation.db.
The PostgreSQL backend stores data in PostgreSQL when backend/server_postgres.py is running and DATABASE_URL is correctly configured.
If no backend API is running, the app falls back to browser localStorage.

Cloud status:
This project is not saving to cloud by default.
Right now it saves to the user's system/local machine unless PostgreSQL is configured on a remote cloud server and DATABASE_URL points to that cloud database.

PostgreSQL mode:
Run PostgreSQL, create the database mpro_reconciliation, set DATABASE_URL, then run:
python backend/server_postgres.py

Example:
$env:DATABASE_URL="postgresql://postgres:your_password@localhost:5432/mpro_reconciliation"
python backend/server_postgres.py

Frontend:
Open index.html directly or serve the folder locally. During local testing it can be opened at:
http://127.0.0.1:8000/index.html
