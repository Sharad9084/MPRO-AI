TAG-mPRO Project Explanation
============================

Project name:
TAG-mPRO - Marketing Performance Reconciliation and Optimisation

Purpose:
This application helps an auditor validate marketing campaign spending by uploading multiple business files, converting them into structured tables, and reconciling values across Program, PR, PO, media schedule, agency invoice, third-party invoice, and monitoring data.

Current application flow:
1. Entry screen with TAG-mPRO dashboard preview
2. Role selection screen
3. Auditor sign in or account creation
4. Accounting workspace
5. New or existing reconciliation selection
6. Compact source-file upload area
7. CRM-style table workspace with search, view tabs, filter chips, configurable columns, and summary counters
8. Reconciliation issue review
9. Save the reconciliation case

Deployment and sign-in behavior:
The frontend can work as a static deployed site.
If a backend API is configured and reachable, sign in uses the backend session token and saved cases can be stored in the database.
If a local or direct-file frontend cannot reach a backend API, the demo auditor account can sign in locally and saved cases stay in browser localStorage.
Users can also create browser-local accounts for local/direct-file testing. Local account passwords are stored as PBKDF2-SHA256 hashes using the browser Web Crypto API.
This avoids showing a local server-start requirement to deployed users.
Vercel can host the frontend and the shared backend together by using Python Functions under the api/ folder.
Vercel will not run backend/server.py as a continuously running Python server; instead the deployed API routes call the PostgreSQL backend logic per request.
Database-backed multi-user storage requires a hosted PostgreSQL connection string in the Vercel project environment variables. Supported names are DATABASE_URL, POSTGRES_URL, POSTGRES_PRISMA_URL, POSTGRES_URL_NON_POOLING, NEON_DATABASE_URL, and SUPABASE_DB_URL.

API configuration:
The frontend checks window.TAG_MPRO_API_BASE or localStorage key mpro.apiBase first.
For localhost or direct file usage, it defaults to http://127.0.0.1:8787.
For deployed domains, it defaults to the same-domain /api Vercel Functions.

Security:
Backend accounts are created through the API when the API is running.
Passwords are not stored as plain text. The backend stores a password salt and PBKDF2-SHA256 password hash.
Password rules require at least 8 characters, uppercase, lowercase, number, and special character.
Repeated failed sign-in attempts lock the account temporarily.
Backend sign in creates a server-side session token.
Sign off revokes the active session token when one exists.
Local demo sign in is only a browser-local fallback for static deployments.

Logical tables in the app:
1. Program / Campaign
   Top-level campaign data such as budget, campaign ID, campaign name, brand, advertiser, and manager.

2. PR
   Purchase requisition level data. One campaign/program can have multiple PR numbers.

3. PO
   Purchase order data. One PR can have multiple PO numbers. Total PO amount should not exceed PR amount.

4. Media Schedule
   Planned campaign execution data such as channel, program, date, air time, spots, rate, and planned amount.

5. Agency Invoice
   Agency invoice data with invoice number, invoice date, PO/PR references, campaign, channel, spot, and cost values.

6. 3rd Party Invoice
   External vendor invoice data. For TV this can mean broadcaster invoice, for print publisher invoice, and for other media a third-party vendor invoice.

7. 3rd Party Monitoring Report
   Proof/monitoring data. For TV this can be AdEx or BARC, for print tear sheet copies, and for digital platform, third-party monitoring, or DCM reports.

8. Reconciliation
   Calculated table that compares the above tables and highlights missing data, amount mismatches, monitoring gaps, and budget hierarchy issues.

Important business hierarchy:
Program/Campaign is the top level.
One Program/Campaign can have many PR numbers.
One PR number can have many PO numbers.
All PR amounts together should stay within the campaign/program budget.
All PO amounts under a PR should stay within that PR amount.

Current reconciliation checks:
- PR data missing
- PO missing
- Media schedule missing
- Agency invoice missing
- 3rd party invoice missing
- Monitoring missing
- PR amount exceeds program budget
- PO amount exceeds PR amount
- PO vs agency amount mismatch
- Schedule vs 3rd party invoice amount review
- Agency vs 3rd party invoice amount review

Table behavior:
The Columns button lets the user show or hide columns for the active table.
The grid supports sorting, resizing, and reordering columns.
Reconciliation and combined views are read-only table views.
Source tables remain editable where manual correction is useful.
The search field filters visible rows across table values.
Active filters appear as removable filter chips.
Filters are driven by field aliases, so uploaded files can use common variants such as PR No, PO No, Invoice No, Campaign Manager, Program Name, POP, and Expense Monitoring.

Storage behavior:
If the backend API is reachable and the user has a server session token, Save sends data to the backend.
The local SQLite backend stores data in backend/mpro_reconciliation.db.
The PostgreSQL backend stores data in PostgreSQL when backend/server_postgres.py is running and DATABASE_URL is configured.
If no backend API is configured or reachable, Save stores cases in browser localStorage.

Cloud status:
This project is not saving to a cloud database by default.
Vercel deployment saves to PostgreSQL when a supported PostgreSQL environment variable is configured and the deployed /api functions are healthy.
If PostgreSQL is missing or unreachable on a deployed Vercel domain, sign-in reports a cloud database error.
Browser-local accounts and cases are device/browser-specific. Sharing one Vercel link only shares a database between users when the Vercel project has a working hosted PostgreSQL connection string.
For deployed Vercel domains, sign-in now reports a cloud database error when PostgreSQL is missing instead of presenting local storage as shared storage.

PostgreSQL mode:
Run PostgreSQL, create the database mpro_reconciliation, set DATABASE_URL, then run:
python backend/server_postgres.py

Example:
$env:DATABASE_URL="postgresql://postgres:your_password@localhost:5432/mpro_reconciliation"
python backend/server_postgres.py
